Question 2: 

There does exist a sequence for k=3 such that OPTFF incurs strictly fewer misses than LRU or FIFO. One such sequence is (6 3 1 3 5 3 7 3 7 9 0 1 4 2 6 8 3 6 5 2 4 6 2 5 8 2 3 6 1 0 9 8 3 2 4 7 5 0 4 8 5 9 2 0 3 4 5 0 7 8 8 0 3 2 1 9 8 3 4 6 4 7 5 1 3 0 3 4 8 5 7 2 6 1 0 1 0 9 8 5 6 7 9 0 1 2 3 3 4 5 7 8 0 3 4 1 8 8 0 2). When running it in the program, the number of misses for OPTFF was 61, whereas FIFO had 87 and LRU had 85 (in the input and output folders of the project, this example is listed under Question2_CacheSize3_Input/Output). The reason this is the case is because OPTFF takes into account more possibilities for when the numerical id could be used again in comparison to LRU and FIFO. For example, OPTFF only evicts the element in a cache that will be accessed the latest of the elements in the cache. This means that it won't be used anytime soon, so the odds of hit occurring goes up. Breaking it down for this k example, since the cache can only hold three elements at one time and there are ten total elements, the odds of a hit occurring is 30%. For the number of hits to rise, the ideal scenario is that the number you add will occur before the number you evict, and even if it doesn't the odds of a hit occurring remains at essentially 30%. But, if it is the case the number you add does lead to a hit sooner, you then have odds of 33% being a hit because the number of possible numerical ids went from 10 to 9 and number of possible hits is 3/9. In comparison, LRU and FIFO remain at this 30% since they don't run the possibility that the very next number to check could be the one that was just evicted. What this all essentially boils down to is the OTPFF is better able to plan for the possibility that the evicted numerical id is less likely to be a hit over the newly inserted, and as such has a higher rate of success in landing hits (and therefore fewer misses) than either LRU and FIFO since they aren't designed to anticipate this possibility.

Question 3: (How to structure exchange argument taken from here: https://leetcopilot.dev/leetcode-pattern/greedy/exchange-argument-explained)

- Let G represent OPTFF/Belady's Farthest-in-Future algorithm and A be any offline algorithm that knows the full request sequence. For any fixed sequence m, we prove Misses(G) ≤ Misses(A)
- If G = A, then we know that the Misses(G) is no larger than Misses(A) and is therefore optimal.
- If G ≠ A, let d be the point at which they differ. Up until d, G and A have identical caches. 
	- G evicts the item currently in the cache whose next request occurs farthest in the future (we'll call it f)
	- Since A can't have an identical cache as G at d, it must evict some other item besides f
- We construct a new algorithm called A' that takes on all values of A besides the value that differed between G and A, which we will call A_d. We replace A_d  with the item from G at occurrence d, which we will call G_d.
	- To simplify, the new algorithm A' takes on A ({a_1,…,a_d,…,a_m}), but replaces A_d with G_d, thus A' becomes {a_1,…,g_d,…,a_m}
- Since we evict G_d before A_d in A', we know that if we were to replace A_d with G_d  then the number of misses cannot increase. Thus Misses(A') ≤ Misses(A).
	- For every instance where G differs from A at occurrence d, this condition will hold and therefore G, up to d, is as optimal as A
	- For each difference, A' can replace G_d  and A_d where necessary
- We repeat this process until each disagreement between the algorithms is eliminated.  
	- We can say that A' is as optimal as A with each difference being stored in A' as overall Misses(A') ≤ Misses(A)
	- Therefore, we can say that Misses(G) ≤ Misses(A), thus G is optimal.
